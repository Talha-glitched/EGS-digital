# 09 Visual Audit And Page System

Use this with `EGS_WEBSITE_CONTENT_BIBLE.md`, `DESIGN_PHILOSOPHY.md`, and the individual page specs.

This is not a color or font redesign. Keep the existing warm editorial palette, serif/sans pairing, quiet borders, generous spacing, and premium feel. The work now is composition: make each page communicate through images, stats, cards, process artifacts, and proof before long copy.

## Visual North Star

EGS should feel like a professional project notebook with Apple-level calm:

- image-led,
- precise,
- tactile,
- spacious,
- highly readable,
- operational without feeling corporate,
- creative without becoming vague.

Think: a serious production file opened on a beautiful desk. Each page should feel like a different chapter in the same notebook.

## Online Reference Takeaways

Use these as inspiration, not as templates to copy.

- Apple product pages: one idea per section, large imagery, short copy, strong product proof, and generous space. Reference: `https://www.apple.com/iphone/`
- Linear: calm operational clarity, compact cards, product/system visuals, and disciplined hierarchy. Reference: `https://linear.app/`
- Notion: notebook-like structure, friendly modular blocks, simple diagrams, and approachable information density. Reference: `https://www.notion.com/product`
- Stripe: complex business information made scannable through modular sections, proof blocks, and structured visual systems. Reference: `https://stripe.com/`

## Current Visual Audit

### What Is Already Working

- The type direction feels distinctive and premium.
- The warm paper background supports the notebook idea.
- The client/project images give the site real material.
- The page-specific accent colors are useful.
- The service pages already have different personalities.
- The home page hero image collage is a good direction.

### What Is Not Working Yet

- Above-the-fold service page headlines are too pale during the reveal state. Hero content must be readable immediately.
- Some service pages are too text-led before the project image does enough work.
- Pages currently feel related, but Exhibitions, Events, and Fitouts use similar hero formulas.
- Some visuals are decorative when they should explain proof.
- Several sections explain the strategy instead of showing the project artifact.
- Case studies and proof need more "file" structure: tags, metrics, captions, and physical evidence.

## Hard Visual Rules

- First-viewport text must never start hidden, washed out, or dependent on scroll reveal.
- Every page needs one dominant visual idea, not only a hero image.
- Every proof story needs a visible metric, time window, client/event name, and physical object.
- Do not use the same hero layout on every service page.
- No page should rely on a long paragraph to explain why EGS is good.
- Every long section should become cards, a stepper, a ledger, a timeline, an annotated image, a proof strip, or a checklist.
- Use captions as evidence, not decoration.
- Keep cards squared or lightly rounded. Avoid soft SaaS blob/card excess.
- Maintain visual silence: thin rules, whitespace, careful alignment, and limited accent color.

## Shared Visual Components

These should exist across pages so the site feels familiar.

### 1. Proof Chip

Small inline stat or fact.

Examples:

- `33 locations`
- `Midnight-before 6am`
- `10 hours before showtime`
- `200 sqm stand`
- `5-6 added chillers`

Use these near images, in hero proof strips, and on case cards.

### 2. Project File Card

A compact case-study card.

Fields:

- service tag,
- client/project name,
- year/event/place,
- one metric,
- one physical thing EGS did,
- link to case anchor.

This should feel like a production note, not a blog card.

### 3. Notebook Stepper

Used for process.

Visual style:

- thin connecting line,
- 5-6 compact steps,
- one noun heading per step,
- one sentence max,
- optional small icon or number.

### 4. Annotated Image

Use on service pages.

Image plus small labels pointing to:

- stage extension,
- display counter,
- chillers,
- signage,
- guest flow,
- mall access,
- QA/QC,
- handover.

Labels should be factual, not decorative.

### 5. Stat Poem

Use for HCT and Sadia.

Large serif numbers stacked or staggered:

```text
7 ceremonies
4,500 graduates
13,500 guests
```

or:

```text
33 locations
13 vehicles
midnight to before 6am
```

### 6. Risk Checklist

Use where buyers are anxious.

Examples:

- access windows,
- material availability,
- venue rules,
- display needs,
- QA/QC,
- handover timing.

Keep it visual and short.

## Page-By-Page Visual Direction

## Home

### Role

The front cover of the notebook.

### Visual Idea

An editorial project-board: a large hero statement, real image collage, client marquee, service cards, and proof files.

### Above The Fold

Use:

- left: H1 and short copy,
- right: asymmetrical image collage,
- bottom: three proof chips.

Recommended proof chips:

- `Founded 2010`
- `25+ big projects/year`
- `Dubai / UAE production`

### Key Visual Sections

1. `Service routing`: four image cards.
2. `Proof files`: horizontal project-file cards.
3. `How EGS works`: stepper, not a paragraph.
4. `HCT anchor`: large image plus stat poem.
5. `Final CTA`: contact strip with deadline/location/scope.

### Make It Different

Home should be the only page with a broad collage of multiple services. Other pages should focus on one service world.

## Exhibitions

### Role

The technical floor plan page.

### Visual Idea

An annotated stand file: render-to-site pressure, product display zones, opening-day readiness, and last-minute adaptation.

### Above The Fold

Use:

- large stand image,
- H1 beside or over it,
- proof chip row: `200 sqm`, `10-12 hours`, `Global Health Riyadh`, `Ultrasound display`.

### Key Visual Sections

1. `Stand anatomy`: annotated image/card showing display counter, demo zone, graphics, storage, product area.
2. `Opening-day pressure`: risk checklist.
3. `Philips proof`: image-led card.
4. `Kazakhstan proof`: second proof card with chillers.
5. `Build process`: notebook stepper.

### Make It Different

Use more technical linework, floor-plan cues, labels, measurements, and section cuts. This page should feel like a stand file.

## Events

### Role

The run-of-show page.

### Visual Idea

A ceremony cue sheet meets a flagship case study.

### Above The Fold

Use:

- HCT ceremony image,
- H1,
- large stat poem:
  - `7 ceremonies`
  - `4,500 graduates`
  - `13,500 guests`

### Key Visual Sections

1. `HCT scale`: large stats with image.
2. `Fujairah 10-hour change`: urgent project file card.
3. `Run-of-show`: cue-sheet inspired visual, but only using confirmed/general labels.
4. `What has to be ready`: stage, branding, guest movement, room readiness, handover.
5. `Process`: backwards-from-showtime stepper.

### Make It Different

Use rhythm, time, showtime, cues, and stage imagery. This page should feel like an event production notebook.

### Important Guardrail

Remove or replace any visible claim like `0 missed cues` unless verified. The content bible says not to say that.

## Retail

### Role

The rollout control room.

### Visual Idea

A logistics map/checklist: locations, vehicles, access windows, installers, QA/QC.

### Above The Fold

Use:

- retail/chiller/mall image,
- H1,
- proof chip row:
  - `33 Carrefour locations`
  - `13 vehicles`
  - `midnight-before 6am`
  - `QA/QC supervisors`

### Key Visual Sections

1. `Sadia control board`: stat poem plus route/vehicle/checklist visual.
2. `What gets installed`: asset grid.
3. `Rollout process`: route-style stepper.
4. `Failure points`: short checklist.
5. `Money Kicks`: small credibility strip, not major proof.

### Make It Different

Use map/list/dispatch language visually: location rows, checkmarks, vehicle numbers, time window, QA status.

## Fitouts

### Role

The material and handover page.

### Visual Idea

A refined interiors notebook: surfaces, signage, joinery, circulation, handover notes.

### Above The Fold

Use:

- large interior/showroom image,
- two detail crops,
- H1,
- capability chip: `Branded interiors · signage · joinery · handover-ready spaces`.

### Key Visual Sections

1. `Room anatomy`: annotated image showing signage, joinery, reception/display, circulation.
2. `What EGS builds`: image-led capability grid.
3. `Fitout process`: stepper.
4. `Daily-use checklist`: movement, visibility, storage, wear points, handover.
5. `Proof/credibility`: small client/image strip unless full details are confirmed.

### Make It Different

Use close-ups, material labels, before/after feeling, and handover notes. This should feel quieter and more architectural than Retail or Events.

## Case Studies

### Role

The proof archive.

### Visual Idea

A project file index.

### Above The Fold

Use:

- H1,
- short paragraph,
- proof-card grid,
- one featured case.

### Key Visual Sections

1. `Proof index`: cards with tags and metrics.
2. `Featured case`: HCT or Sadia with image/stat treatment.
3. `Case sections`: anchored files with short labels.
4. `Source notes`: small captions, not large paragraphs.
5. `Final CTA`: "Which pressure looks closest to yours?"

### Make It Different

Use archive/index language visually: tabs, case numbers, tags, dividers, file cards, sticky side navigation if simple.

## Contact

### Role

The intake desk.

### Visual Idea

A clean brief form plus direct contact.

### Above The Fold

Use:

- H1,
- one sentence,
- email/phone/WhatsApp buttons,
- four brief cards visible immediately or just below.

### Key Visual Sections

1. `Choose your brief`: four cards.
2. `Form`: grouped fields, not one long block.
3. `What to include`: checklist.
4. `What happens next`: five-step strip.
5. `Urgent proof`: small proof chips only.

### Make It Different

This page should be the most practical and least poetic. It should feel like an elegant operations form.

## Current Site Fixes Before Content Implementation

### Priority 1: Fix Hero Readability

Current service hero screenshots show major hero text faded out. Above-the-fold hero content should not use `.reveal` unless the page initializes it immediately. H1, lede, CTA, first image, and proof chips must be readable on first paint.

### Priority 2: Remove Unverified Visual Claims

Remove or replace:

- `0 cues missed`,
- unverified stand sizes,
- reuse rates,
- cost-per-sqm improvements,
- NPS/lead lift,
- unconfirmed year-by-year Philips timeline,
- any detailed scope not in the Content Bible.

### Priority 3: Make Each Page Visually Distinct

- Home: broad project board.
- Exhibitions: annotated stand/floorplan.
- Events: run-of-show/cue sheet.
- Retail: rollout control board.
- Fitouts: material/handover notebook.
- Case Studies: proof archive.
- Contact: intake desk.

### Priority 4: Make Proof More Visual

Every key proof story should be reduced into a visual atom:

- `client/project`,
- `place/event`,
- `year`,
- `time pressure`,
- `physical work`,
- `result`.

Use these atoms as cards, captions, chips, and anchor links.

## Implementation Rule For You

When implementing content yourself, build the page in this order:

1. hero,
2. proof chip or trust strip,
3. dominant visual section,
4. service capability cards,
5. process visual,
6. proof/case cards,
7. FAQ,
8. CTA.

Then remove 20-30% of the text. If the page still makes sense, the visual system is doing its job.
